## Source Registry Design

### Data File

`main/data/source_registry.json` — flat array of source records (not a bibliography, not a citation list).

### Required Fields per Record

| Field | Purpose |
|-------|---------|
| `id` | Stable unique identifier |
| `slug` | URL-safe unique slug |
| `label` | Human-readable source name |
| `source_class` | One of 7 governed source classes |
| `authority_role` | What authority this source provides |
| `claim_scope` | What the source can and cannot establish |
| `verification_posture` | Confidence level and conditions |
| `related_page_types` | Which page types this source supports |
| `boundary_notes` | Explicit limitations on use |

### Source Classes (7 required)

| Class | Description |
|-------|------------|
| `institutional-space-agency` | NASA, ESA, JAXA, ISRO — primary institutional references |
| `government-policy` | Government-commissioned studies, policy documents |
| `academic-research` | Peer-reviewed academic publications |
| `technical-standard` | IEEE, ITU — standards and regulatory references |
| `industry-consortium` | Industry and consortium research publications |
| `historical-primary` | Historical origin documents and programme records |
| `methodology-internal` | Internal site governance — not for external claims |

## /sources/ Hub Upgrade

The /sources/ hub is now generated by `generate_sources_hub_html()` in build.py via `seedSource: "sources_hub"` routing. It includes:

1. **Status panel** — total governed sources, source classes, claim boundary indicator
2. **Governance explanation** — what the registry is and how it works
3. **How sources are used** — tier system explanation, claim category application
4. **Claim boundary explanation** — what is and is not claimed about SBSP status
5. **Source class jump navigation** — anchor links to each of the 7 class sections
6. **Source class sections** — all 26 records organized by class with label, authority role, claim scope, and boundary notes
7. **Footer links** — to methodology, framework, feasibility, strategic importance, global programs, programs, glossary, questions, and technology

## Source Governance Validator (15th)

`main/scripts/validate_sources_registry.py` checks:

- Stable IDs — no duplicates
- Unique slugs — no duplicates
- All 9 required fields present and non-empty
- `source_class` must be one of 7 allowed values
- `claim_scope` minimum 60 characters
- `boundary_notes` minimum 40 characters
- No banned vague authority language in any text field
- All 7 source classes must be represented (minimum coverage requirement)

### Banned Vague Authority Phrases

- "proves that"
- "confirms that sbsp works"
- "confirms commercial viability"
- "established fact"
- "widely accepted truth"
- "definitively shows"
- "conclusively proves"
- "industry consensus that sbsp is ready"

## Architecture Notes

### Why a new validator rather than strengthening validate_sources.py

`validate_sources.py` checks sourceRef cross-references and sourceBoundary discipline across pages. `validate_sources_registry.py` checks the governance of the source_registry.json file itself — schema integrity, coverage of all source classes, field completeness, and banned language. These are distinct concerns warranting distinct validators. Validator count increases from 14 to 15.

### Why no source subpages

Creating individual source subpages would create 26 additional thin pages requiring internal links, sitemap entries, and cross-references to justify them. The governed hub model captures all necessary source intelligence without creating thin content. Source profiles would only be justified when a source requires extensive documentation that cannot fit in a registry record — that is a future decision point.

### Sitemap count

177 pages — unchanged. The /sources/ hub is already included. No new pages were added.
