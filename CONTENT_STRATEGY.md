# Content Strategy: Space-Based-Solar-Power.com

## 1. Purpose

This document defines the content strategy for Space-Based-Solar-Power.com.

The asset is not being built as a general blog, a news site, a renewable-energy explainer, an affiliate property, or a thin SEO website.

It is being built as a sovereign-grade, multilingual, source-disciplined, interface-driven reference asset for Space-Based Solar Power and Orbital Energy Infrastructure.

The content strategy exists to answer one central question:

> What content will make Space-Based-Solar-Power.com a trusted reference for governments, investors, journalists, researchers, companies, engineers, analysts, and future strategic buyers?

The answer is not “many articles.”

The answer is a governed knowledge system.

---

## 2. Governing Content Principle

The governing content principle is:

> Every page must either define the category, clarify the framework, explain a constraint, support a strategic audience, strengthen internal linking, improve source trust, enable a tool, support multilingual authority, or serve buyer logic.

No page may exist merely because a keyword exists.

No article may exist merely because a topic is trending.

No route may exist merely because a normal website would have that route.

Content must make the asset more authoritative, more useful, more defensible, more indexable, more trusted, or more valuable.

---

## 3. Core Content Thesis

Space-Based-Solar-Power.com does not treat SBSP as a narrow technical curiosity.

It treats SBSP as a future infrastructure category.

The content system must consistently frame SBSP across the following infrastructure chain:

```text
Sunlight
→ Orbital Energy Collection
→ Solar Power Satellite / Orbital Platform
→ Wireless Power Transmission
→ Atmospheric Crossing
→ Rectenna / Receiving Infrastructure
→ Terrestrial Grid / Remote Power / AI Load / Defense Resilience
→ Lunar and Space-Industrial Extension
```

This chain is the content spine of the asset.

Every major page must connect back to this spine.

## 10. Internal Linking Doctrine

Every page must connect to the asset’s framework.

A page should link upward, sideways, and forward.

### Upward Links

Every page should link to at least one of:

```text
/framework/
 /methodology/
 /sources/
 /tools/
```

### Sideways Links

Every page should link to related pages in the same cluster.

Examples:

- `wireless-power-transmission` links to `microwave-power-beaming`, `laser-power-transmission`, and `rectenna-infrastructure`.
- `sbsp-economics` links to `launch-economics`, `feasibility-and-constraints`, and `orbital-energy-constraint-matrix`.
- `sbsp-and-ai-energy-demand` links to `strategic-importance`, `grid-resilience`, and `energy-demand-of-intelligence-infrastructure`.

### Forward Links

Every page should point the user toward a next deeper action:

- read the methodology;
- examine sources;
- use a tool;
- compare concepts;
- review a claim boundary;
- explore the program registry.

No page may be an isolated content island.
