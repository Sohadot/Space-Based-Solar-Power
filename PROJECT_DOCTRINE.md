# Project Doctrine: Space-Based-Solar-Power.com

## 1. Purpose

This doctrine governs the development, publication, localization, monetization, and technical operation of Space-Based-Solar-Power.com.

The project must not be treated as a generic website, news blog, affiliate shell, speculative investment page, or thin SEO property.

It is a sovereign-grade multilingual category reference asset for Space-Based Solar Power and Orbital Energy Infrastructure.

## 2. Content Mandates

All content must be written in a rigorous, analytical, source-disciplined style.

No technical, economic, scientific, engineering, regulatory, policy, investment, or physical claim may be published without either:

1. a verifiable source entry;
2. a clearly marked conceptual boundary;
3. or an explicit statement that the claim is strategic interpretation rather than established fact.

Generalist articles about solar panels, climate change basics, green living, or ordinary renewable energy are prohibited unless directly connected to SBSP mechanics, economics, constraints, infrastructure, or strategic implications.

Every public page must serve a defined role in the asset architecture.

No page may exist merely to capture keywords.

## 3. Source-Control Rule

The asset must distinguish between:

- source-verified statements;
- institutional summaries;
- technical explanations;
- strategic interpretation;
- conceptual framing;
- unresolved future scenarios.

Speculation is allowed only when clearly labeled.

Speculation must never be presented as established fact.

## 4. Structural Integrity

No broken user journeys are allowed.

The production build must not include:

- placeholder text;
- empty pages;
- coming-soon pages;
- dead navigation links;
- fake article archives;
- unverified localized pages;
- sitemap entries for unpublished routes;
- language switcher links pointing to missing pages.

If a page is not complete, validated, internally linked, and approved for publication, it remains outside the production build.

## 5. Technical Doctrine

The repository is the single source of truth.

The architecture must remain lightweight, static, auditable, and source-controlled.

Permitted approaches include:

- native HTML/CSS/JS;
- Python-based static generation;
- approved static build workflows.

No heavy external framework, database backend, plugin dependency, third-party rendering layer, or uncontrolled build dependency may be introduced without a documented architectural decision.

## 6. Measurement and Tag Governance

Approved measurement infrastructure is limited to:

- Google Search Console;
- Google Analytics 4;
- Google Tag Manager.

GTM may be used as the controlled measurement container.

Duplicate analytics tags, uncontrolled third-party pixels, retargeting scripts, and random marketing trackers are prohibited unless explicitly approved through a documented decision.

All measurement scripts must be reviewed under the security policy.

## 7. Monetization Doctrine

The asset must not be monetized in a way that weakens institutional trust or buyer perception.

Programmatic ad networks are prohibited during the authority-building phase.

Future monetization may include:

- premium research briefs;
- strategic reports;
- institutional sponsorship;
- research partnerships;
- controlled lead generation;
- high-quality educational resources.

Any monetization layer must reinforce the asset’s authority rather than extract short-term value from it.

## 8. Security Doctrine

Security headers must be enforced at the hosting or edge layer.

Cloudflare may be used for DNS, TLS, WAF, and edge security controls.

GitHub remains the source of truth for development and publication workflow unless a future documented decision changes this architecture.

No random external scripts, unreviewed embeds, unsafe inline JavaScript, mixed-content assets, or uncontrolled third-party resources are allowed.

## 9. Multilingual Doctrine

Localization is not translation.

Raw, unreviewed machine translation is prohibited from production.

Every localized page must preserve:

- technical accuracy;
- source boundaries;
- institutional tone;
- localized strategic emphasis;
- correct language metadata;
- valid canonical URL;
- reciprocal hreflang mapping;
- same-language internal links where available.

Slugs remain standardized to the English source layer unless a future documented decision changes the URL policy.

Example:

`/ar/what-is-space-based-solar-power/`

The language directory changes. The core slug remains stable.

## 10. Strategic Tools Doctrine

Strategic tools are allowed and encouraged when they strengthen the asset’s reference value.

Tools must function as decision-support and reference instruments, not speculative prediction engines.

No tool may publish unsupported technical, economic, regulatory, investment, or policy claims.

Qualitative tools must disclose their classification logic.

Numerical tools must disclose assumptions, sources, ranges, limitations, and methodology version.

The first approved tools layer prioritizes qualitative and source-disciplined frameworks before numerical calculators.

## 11. Hreflang Doctrine

A localized page may only be deployed when its hreflang map includes the published and validated language variants of the same content node.

Hreflang must not point to drafts, missing pages, redirects, placeholders, or unpublished variants.

Each localized variant must use a self-referencing canonical URL.

## 12. Quality Gate

No public deployment is acceptable unless the quality gate passes.

At minimum, validation must cover:

- internal links;
- external link handling;
- SEO metadata;
- canonical URLs;
- hreflang integrity;
- language switcher integrity;
- sitemap entries;
- robots.txt;
- placeholder detection;
- source-claim boundaries;
- asset loading;
- responsive layout;
- accessibility basics.

Failure of the quality gate blocks publication.
